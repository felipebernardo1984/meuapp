IMPORTANTE:

* NÃO alterar lógica atual de check-in
* NÃO remover funcionalidades existentes

# 1. PADRONIZAR INTEGRAÇÃO DO ALUNO (CRÍTICO)

Garantir que cada aluno possua um campo único e confiável:

student:

* integrationType: 'wellhub' | 'totalpass' | 'none'
* plan: string (ex: TP1, TP2, GP1...)

IMPORTANTE:

* Esse campo deve ser a única fonte de verdade
* NÃO usar lógica espalhada ou implícita

---

# 2. AJUSTAR E GARANTIR ESTRUTURA POR MODALIDADE

Cada modalidade deve conter:

Wellhub:

* wellhubPlanoMinimo
* wellhubValorCheckin

TotalPass:

* totalpassPlanoMinimo
* totalpassValorCheckin

IMPORTANTE:

* NÃO misturar valores
* NÃO usar valor único por modalidade

---

# 3. CRIAR FUNÇÃO CENTRAL DE CÁLCULO (OBRIGATÓRIO)

Criar um service central (financeService):

Função obrigatória:

getValorCheckin(modalidade, aluno):

Regras:

* se aluno.integrationType === 'wellhub'
  → usar wellhubValorCheckin
* se aluno.integrationType === 'totalpass'
  → usar totalpassValorCheckin
* se 'none'
  → retornar 0 ou valor padrão definido

Essa função deve ser a ÚNICA responsável pelo cálculo

---

# 4. VALIDAR PLANO MÍNIMO (REGRA DE NEGÓCIO)

Antes de permitir check-in:

* verificar se plano do aluno atende o plano mínimo da modalidade

Exemplo:

* aluno TP1 não pode acessar modalidade que exige TP2

Ação:

* bloquear OU alertar (sem quebrar fluxo atual)

checkinFinanceiro:

IMPORTANTE:

# 6. GERAR RECEITA AUTOMATICAMENTE

Após cada check-in:

1. identificar aluno
2. identificar modalidade
3. calcular valor via financeService
4. salvar em checkinFinanceiro

IMPORTANTE:

* garantir que cada check-in gere apenas UM registro financeiro
* evitar duplicidade
